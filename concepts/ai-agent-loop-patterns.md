## 배경

AI 에이전트가 단순히 한 번 생성하고 끝내는 것이 아니라, 스스로 반복하며 품질을 높이는 루프 패턴이 주목받고 있다. 대표적인 두 패턴인 **Reflection Loop**와 **Ralph Loop**는 목적은 비슷하지만 설계 철학이 다르다.

---

## Reflection Loop

LLM이 자신의 출력을 비판하고 개선하는 반복 패턴. "생성 → 반성 → 개선"이 핵심이다.

```
Generate → Reflect (자연어 비판) → Refine → Reflect → ...
```

피드백은 자연어다. "이 답변은 근거가 부족하다", "3번 항목에 논리적 비약이 있다"처럼 모델이 스스로 판단해 텍스트로 기록하고, 그 피드백을 다음 생성에 주입한다.

종료 조건이 주관적이다. Reflector가 "이 정도면 충분하다"고 판단하거나, 최대 반복 횟수에 도달하면 멈춘다.

**대표 연구:**
- **Reflexion** (Shinn et al., 2023) — 실패 경험을 언어 피드백으로 저장해 다음 시도에 활용. RL 없이 언어 강화만으로 에이전트 성능 향상을 보임
- **Self-Refine** (Madaan et al., 2023) — 동일 모델이 Generator와 Reflector를 모두 담당. 외부 데이터 없이 반복 개선

---

## Ralph Loop

AI가 코드를 쓰고 테스트하고 고치는 자율 루프. 사람 개입 없이 완료 조건(테스트 통과, 타입 에러 0)이 충족될 때까지 반복한다.

```
작업 → 테스트/린터 실행 → FAIL → 가드레일 기록 → 재시도 → ... → PASS → 종료
```

피드백은 객관적 신호다. 자연어 비판 대신 "테스트 3개 실패", "타입 에러 2개"처럼 명확한 결과가 루프를 구동한다.

메모리를 대화 컨텍스트에 두지 않고 **파일 시스템**에 둔다. 매 반복마다 컨텍스트 창을 새로 시작해도, 코드베이스와 가드레일 파일에 상태가 보존된다. 가드레일 파일은 "이 접근은 실패했다"는 표지판 역할을 해 같은 실수 반복을 방지한다.

Claude Code의 stop hook으로 구현된다. Anthropic이 공식 플러그인(`ralph-wiggum`)을 제공하며, `/ralph-loop "목표"` 명령으로 시작한다.

## 비교

| | Reflection Loop | Ralph Loop |
|---|---|---|
| **도메인** | 텍스트, 추론, 코드 등 범용 | 코드 특화 |
| **피드백 형식** | 자연어 비판 | 객관적 신호 (테스트/타입/린터) |
| **메모리** | 컨텍스트 창 (또는 외부 저장소) | 파일 시스템 + git 이력 |
| **종료 조건** | 주관적 품질 판단 | 명확한 조건 (에러 0개, 테스트 PASS) |
| **구현 레벨** | 모델 설계 패턴 (논문/연구 수준) | 도구 수준 (Claude Code 플러그인) |
| **컨텍스트 지속성** | 대화 히스토리에 의존 | 반복마다 새 컨텍스트 가능 |

## 언제 무엇을 쓸까

**Reflection Loop가 적합한 경우:**
- 정답이 주관적이거나 다차원적인 품질이 필요할 때
- 복잡한 추론·글쓰기처럼 한 번에 완성이 어려운 작업
- 코드 이외 도메인 (요약, 계획, 분석 등)

**Ralph Loop가 적합한 경우:**
- 테스트/타입/린터로 자동 검증 가능한 코드 작업
- 반복적·기계적 작업 (리팩토링, 마이그레이션)
- 완료 조건을 명확히 정의할 수 있을 때

## 두 패턴의 공통 원리

두 패턴 모두 "단발성 생성"의 한계를 루프로 극복한다. 차이는 피드백의 성격이다. Reflection은 LLM 판단(주관)을 루프 엔진으로 쓰고, Ralph는 실행 결과(객관)를 루프 엔진으로 쓴다.

실제 에이전트 시스템에서는 두 패턴이 혼합되기도 한다. 코드 생성 후 테스트로 먼저 걸러내고(Ralph 방식), 통과한 코드를 리뷰어 LLM이 다시 비판하는(Reflection 방식) 하이브리드 구조가 대표적이다.
