## 정의

AI 에이전트가 **직접 실행하지 않고**, 실행 가능한 결과물(SQL, 스크립트, 명령어)을 생성해 사람이 검토 후 수동으로 실행하는 설계 원칙.

"AI는 제안하고, 사람이 결정한다."

## 구조

```
데이터 수집 (읽기 전용)
      ↓
AI 탐지 + 분석
      ↓
AI 생성 (fix SQL + rollback SQL)
      ↓
코드 검증 (구조적 오류만)
      ↓
AI 검토 (의미·도메인 판단)
      ↓
리포트 저장 → 사람 검토 → 수동 실행
```

**AI가 절대 하지 않는 것**: DB 직접 수정, 파일 삭제, 외부 API POST 등 되돌리기 어려운 작업

## 왜 필요한가

### 완전 자동화가 위험한 경우

고위험 자동화는 실수의 파급이 크다:

- DB 수정 SQL이 잘못되면 프로덕션 데이터 오염
- AI가 도메인 맥락을 모르고 판단하면 비즈니스 로직 위반
- 자동화가 빠를수록 실수가 대규모로 번짐

### AI의 한계

AI는 패턴 인식과 생성에 강하지만:
- 도메인 고유 예외("이 키워드는 카테고리가 아니라 브랜드")를 모를 수 있음
- 통계적으로 올바른 판단이 특정 비즈니스 상황에서는 틀릴 수 있음
- 롤백 SQL을 생성해도 시뮬레이션과 실제 DB 상태가 다를 수 있음

## 핵심 설계 결정

### 1. rollback SQL 필수 쌍 생성
모든 fix SQL에 대해 rollback SQL을 반드시 함께 생성한다.

```sql
-- fix
UPDATE search_keywords SET keyword_type = 'BRAND' WHERE id = 123;

-- rollback (자동 생성)
UPDATE search_keywords SET keyword_type = 'CATEGORY' WHERE id = 123;
```

사람이 실행하기 전에 "되돌릴 수 있는가"를 확인할 수 있어야 한다.

### 2. 영향도 예측 첨부
SQL 실행 전, AI가 예상 영향도를 리포트에 포함한다:

```
검색 빈도: 월 3,200회
현재 zero-result 비율: 78%
수정 후 예상: 브랜드 필터 적용 → zero-result 해소 예상
```

사람이 "실행할 가치가 있는가"를 판단할 수 있는 정보를 제공한다.

### 3. ESCALATE 케이스 명시
AI가 판단하기 어렵거나 구조적 한계에 부딪힌 케이스는 자동으로 해결하려 하지 않고 사람에게 이관한다.

```
verdict: ESCALATE
이유: 해당 키워드는 브랜드와 카테고리 의미가 혼재합니다.
      담당자가 도메인 판단 후 수동 등록 필요.
```

### 4. 읽기 전용 데이터 수집
데이터 수집 단계에서 DB에 쓰기 작업을 하지 않는다. CSV 스냅샷으로 로컬에 저장해 분석한다.

→ 수집 단계 실패가 프로덕션에 영향을 주지 않음

## 사후 검증 루프와의 관계

비파괴적 원칙 때문에 AI가 직접 실행 후 결과를 확인할 수 없다.

대신 **사후 검증 루프**로 보완한다:
- 사람이 SQL 실행 후 N일 뒤
- 에이전트가 동일 키워드를 재탐지해 이슈가 해소됐는지 확인
- AI가 실행 결과를 간접적으로 학습

## 적용 기준

비파괴적 설계가 필요한 신호:
- 되돌리기 어렵거나 시간이 오래 걸리는 작업
- 도메인 맥락이 코드베이스에 명시되지 않은 경우
- 실수가 사용자 경험에 즉시 영향을 미치는 경우
- AI가 틀렸을 때 책임 소재가 불분명한 경우

완전 자동화를 허용할 수 있는 신호:
- 롤백이 즉시·완벽하게 가능한 작업
- 오류 범위가 작고 격리된 경우 (예: 파일 한 개 생성)
- 사람 검토 비용이 자동화 이득보다 큰 경우
